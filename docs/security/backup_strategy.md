# Стратегия резервного копирования PostgreSQL

## Назначение резервного копирования

Резервное копирование предназначено для восстановления базы данных Slop-Bakery после сбоя PostgreSQL, повреждения данных, ошибочных действий пользователя или несанкционированного изменения записей.

## Защищаемые данные

Защищаются таблицы клиентов, пользователей, ролей, заказов, позиций заказов, счетов, отгрузок, складских остатков, поставок, технологических карт, контроля качества и журнала аудита `audit_log`.

## Угрозы, которые закрывает резервное копирование

Копии помогают восстановиться после удаления или порчи данных, сбоя контейнера или диска, ошибки администратора, некорректного обновления схемы, потери части журнала аудита и последствий атаки на данные.

## Стратегия копирования

В проекте используется логическое резервное копирование PostgreSQL через `pg_dump`. Скрипт `scripts/backup_db.sh` создает SQL-файл, который можно восстановить через `psql` скриптом `scripts/restore_db.sh`.

## Периодичность

Полную резервную копию рекомендуется создавать ежедневно для активной эксплуатации и не реже одного раза в неделю для учебного стенда. Дополнительный логический дамп нужно делать перед изменением схемы БД, ролей, RLS-политик, seed-данных и перед массовой загрузкой данных.

## Создание резервной копии

Переменные подключения задаются через окружение: `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`. Реальные пароли не хранятся в репозитории.

```bash
./scripts/backup_db.sh
```

Дамп сохраняется в каталог `backups` с именем вида `bakery_YYYYMMDD_HHMMSS.sql`.

## Восстановление из резервной копии

Восстановление изменяет состояние целевой БД, поэтому скрипт запрашивает подтверждение. Для автоматического запуска можно использовать `FORCE_RESTORE=1`.

```bash
FORCE_RESTORE=1 ./scripts/restore_db.sh backups/<file>.sql
```

## Проверка восстановления

Не реже одного раза в месяц рекомендуется развернуть последнюю копию в тестовой БД, проверить наличие ключевых таблиц, количество записей в `customers`, `customer_orders`, `invoices`, `shipments`, доступность `audit_log` и выполнение `database/checks/07_test_queries.sql`.

## Хранение резервных копий

Копии нужно хранить вне контейнера PostgreSQL и вне основного рабочего каталога: на отдельном диске, в защищенном сетевом хранилище или закрытом объектном хранилище. Доступ к копиям должен быть ограничен администраторами. Для копий с персональными данными рекомендуется шифрование.

## Ограничения текущей реализации

Скрипты создают логический SQL-дамп и не настраивают непрерывное архивирование WAL. Шифрование, ротация старых копий и отправка во внешнее хранилище пока не автоматизированы и должны быть добавлены при промышленной эксплуатации.
