# Стратегия резервного копирования PostgreSQL

## Назначение

Резервное копирование нужно для восстановления базы данных Slop-Bakery после сбоя, ошибки пользователя, повреждения данных, несанкционированного изменения записей или потери контейнера PostgreSQL.

## Защищаемые данные

Копированию подлежат все прикладные данные PostgreSQL: справочники, пользователи и роли, клиенты, заказы, позиции заказов, счета, отгрузки, складские остатки, производственные партии, результаты контроля качества и журнал аудита `audit_log`.

## Периодичность

Полное резервное копирование рекомендуется выполнять не реже одного раза в неделю, а для активной эксплуатации - ежедневно в нерабочее время.

Логические дампы `pg_dump` рекомендуется выполнять ежедневно, а перед изменением схемы, ролей, политик безопасности или массовой загрузкой данных - дополнительно вручную.

## Создание дампа

Пример команды без сохранения пароля в репозитории:

```bash
PGPASSWORD="$DB_PASSWORD" pg_dump \
  --host="$DB_HOST" \
  --port="$DB_PORT" \
  --username="$DB_USER" \
  --dbname="$DB_NAME" \
  --format=plain \
  --file="backups/bakery_YYYYMMDD_HHMMSS.sql"
```

В проекте для этого предназначен скрипт `scripts/backup_db.sh`.

## Восстановление

Пример восстановления логического дампа:

```bash
PGPASSWORD="$DB_PASSWORD" psql \
  --host="$DB_HOST" \
  --port="$DB_PORT" \
  --username="$DB_USER" \
  --dbname="$DB_NAME" \
  --file="backups/bakery_YYYYMMDD_HHMMSS.sql"
```

В проекте для этого предназначен скрипт `scripts/restore_db.sh`.

## Хранение копий

Резервные копии следует хранить вне основного контейнера и вне рабочего каталога с кодом: на отдельном диске, в защищенном сетевом хранилище или в закрытом объектном хранилище. Доступ к копиям должен быть ограничен администраторами БД. Для копий с персональными данными рекомендуется шифрование и контроль срока хранения.

## Проверка восстановления

Не реже одного раза в месяц нужно разворачивать последнюю копию в тестовой базе, запускать SQL-инициализацию и контрольные запросы, проверять наличие ключевых таблиц, количество записей в `customers`, `customer_orders`, `invoices`, `shipments` и доступность журнала `audit_log`.

## Закрываемые риски

Резервное копирование снижает последствия аппаратного сбоя, повреждения тома Docker, ошибки администратора, случайного удаления данных, некорректного обновления схемы, порчи данных после атаки и потери части журнала аудита.
